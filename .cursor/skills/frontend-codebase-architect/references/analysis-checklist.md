# 前端项目分析清单（read-only）

> 用途：保证分析稳定、可复核、不过度主观。

## 1. 技术栈识别清单

- [ ] 从 `package.json` 识别核心框架与版本线索（React/Vue/Next/Nuxt）。
- [ ] 识别构建工具（Vite/Webpack/Rspack/Rollup）。
- [ ] 识别语言与类型体系（TypeScript/Babel）。
- [ ] 识别样式体系（Tailwind/CSS Modules/Sass/Less/Styled Components）。
- [ ] 识别请求库（axios/fetch 封装）。
- [ ] 识别状态管理（Redux/Zustand/Pinia/Vuex/Context）。
- [ ] 识别代码规范工具（ESLint/Prettier/Stylelint/Husky/lint-staged）。
- [ ] 对无法确认的依赖用途标注“待确认”。

## 2. 目录结构检查清单

- [ ] 说明 `src/` 一级目录职责。
- [ ] 识别页面目录与业务模块边界。
- [ ] 识别通用组件目录与复用层。
- [ ] 识别 `services/api`、`store`、`router`、`hooks`、`utils` 等模块职责。
- [ ] 识别入口文件与初始化链路。
- [ ] 标注目录职责冲突或历史遗留结构。

## 3. 路由检查清单

- [ ] 定位路由注册位置与挂载入口。
- [ ] 识别扁平/嵌套路由模式。
- [ ] 识别懒加载与代码分割策略。
- [ ] 识别鉴权守卫、白名单、重定向逻辑。
- [ ] 识别动态路由与权限路由生成机制。
- [ ] 输出“新增页面 + 新增路由”的标准接入步骤。

## 4. API 封装检查清单

- [ ] 是否存在统一请求实例（axios/fetch wrapper）。
- [ ] 是否统一管理 baseURL、timeout、headers。
- [ ] 是否有请求/响应拦截器。
- [ ] 是否有统一错误处理与错误码映射。
- [ ] Token 注入、刷新、失效回收策略是否明确。
- [ ] API 模块按业务域拆分是否清晰。
- [ ] 输出“新增接口”的标准接入步骤。

## 5. 状态管理检查清单

- [ ] 区分全局状态、页面状态、组件本地状态。
- [ ] 识别 store 组织与切片划分策略。
- [ ] 识别 action/mutation/reducer 的职责边界。
- [ ] 是否有持久化策略（localStorage/sessionStorage/cookie）。
- [ ] 是否存在状态冗余、跨层级重复存储。
- [ ] 输出“新增状态”的标准接入步骤。

## 6. 样式与组件检查清单

- [ ] 识别样式技术栈与约定。
- [ ] 识别组件模式（容器/展示、hooks 复用、组合式）。
- [ ] 识别通用组件与业务组件边界。
- [ ] 识别命名规范与目录组织规范。
- [ ] 识别组件职责过重或重复逻辑。
- [ ] 输出“新增通用组件/页面样式”的推荐做法。

## 7. 风险与技术债检查清单

- [ ] 状态管理是否分散且缺少单一事实源。
- [ ] API 封装是否存在多套并行标准。
- [ ] 目录职责是否混乱。
- [ ] 是否存在过多魔法字符串与硬编码。
- [ ] 是否存在大型组件耦合过重。
- [ ] 是否存在重复逻辑未抽离。
- [ ] 每条风险必须给出代码证据或标注“从当前代码推测”。

## 8. 输出质量检查清单

- [ ] 全文中文。
- [ ] 结论有证据路径。
- [ ] 不确定项明确写“待确认”。
- [ ] 先宏观后微观。
- [ ] 明确“如何新增页面/路由/API/状态/组件/业务模块”。
- [ ] 结果可直接沉淀到 `PROJECT_ARCHITECTURE.md`。
